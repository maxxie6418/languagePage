**发现的问题**

1. docs/考研英语学习站_PRD_v1.0.md 的 ebinhause_records 调度逻辑前后冲突。
   第 4.6 节、5.4 节、8.1 节都写了“标记 mastered 时一次性插入 6 条复习记录”，但第 5.4 节和 8.2 节又写“用户完成当前 stage 后生成下一 stage 记录”（228-231, 673-674）。这两种策略不能同时成立，否则会重复建档。必须二选一：
   要么“预生成全部 stage，仅更新状态”；要么“只生成下一条 pending 记录，按完成逐步推进”。
2. 测验系统缺少核心题库数据模型，导致“真题解析 / 模拟测验 / 错题本”无法真正落地。
   文档定义了 quiz_records、mistakes.question_id，并提供 /api/quiz/questions、/api/quiz/submit 等接口（149-171, 237-241），但没有 questions / question_sets / answers 之类的题库表，也没有题目导入格式、题型字段、标准答案、解析、材料正文、年份/来源等定义。当前 PRD 中，测验模块在交互层存在，在数据层不存在。
3. 多用户场景下，“艾宾浩斯提醒开关”的持久化位置没有定稿，会直接影响实现边界。
   接口里已经定义了 /api/user/ebinhause/settings（224-225），设置页也有该开关（589-593），但 8.3 节却写“存储于用户本地偏好（或后端 user 表扩展字段）”（677-681）。这是架构级分歧，不应在 PRD 里保留“或”。如果支持多设备/多浏览器登录，就应落到后端用户级存储，而不是仅 localStorage。
4. 导入相关接口和页面设计不一致，且缺少“个人数据导入”闭环。
   5.2/5.3 节写了 /api/words/import、/api/knowledge/import（205, 215），5.7 节又写 /api/admin/import/words、/api/admin/import/knowledge（258-259）。两套路由语义重复且权限边界不同。
   另外，设置页有“导入学习数据”按钮（596），但 API 只有 /api/user/export，没有对应 /api/user/import，也没有导入冲突策略（覆盖 / merge / 跳过）说明。
5. 若按当前交互稿实现，部分按钮没有对应的数据模型或接口支撑。
   例如单词展开区里的“加入单词清单”“标记重点”（422）在数据库、API、设置页、导出格式中都没有定义。
   这不是文案细节，而是功能闭环缺失：如果保留这些 CTA，就需要补 custom_word_lists / user_word_flags 一类的数据结构；如果不做，就应从设计稿里删掉，避免 UI 超前于需求。
6. 管理员能力定义和 API / 数据约束不完整。
   角色说明里管理员可“查看用户列表、重置密码、数据备份/恢复”（64-65），页面清单 M11 又出现“删除用户”（816），但 API 只有列表和重置密码（267-268）。
   如果允许删用户，还必须补充：
   - 是否允许删除管理员
   - 是否允许删除最后一个管理员
   - 删除用户时其学习记录是级联删除、软删除还是归档
   - 用户名是否允许复用
7. user_words、user_knowledge、mistakes 等表缺少关键唯一性约束说明，后续很容易出现重复记录。
   例如 105-115 的 user_words 如果没有 (user_id, word_id) 唯一约束，同一用户同一单词可以出现多条状态记录；user_knowledge 同理；mistakes 也需要至少 (user_id, question_id) 的聚合规则。
   这类约束虽然属于实现细节，但对 PRD 的数据设计章节来说属于必要条件。
8. 术语和命名有明显拼写问题，建议在文档层统一。
   ebinhause 不是标准写法，标准术语是 Ebbinghaus；当前表名、接口路径都使用了 ebinhause（136, 221-225）。
   如果这是内部 identifier，也建议现在统一，不然后续数据库、API、前端状态名都会被这个错误拼写固化。

**遗漏项**

1. 缺少“题库内容格式规范”。
   目前只有 words.md 和 knowledge.md 导入格式（620-652），没有真题/模拟题的 Markdown 或 JSON 导入规范。
2. 缺少“作文批改输入界面”定义。
   AI 部分有 /api/ai/essay 和批改结果弹层（250, 813），但没有“作文正文输入、字数统计、提交入口、历史记录”的页面设计。
3. 缺少“恢复 / 备份回滚”定义。
   管理员权限里提到“数据备份/恢复”（65），API 里只有导出全站数据（261），没有恢复接口、恢复前校验、覆盖策略、权限确认流程。
4. 缺少“认证安全边界”补充说明。
   你写了 JWT 放 localStorage（70），但没有定义 CSRF/XSS 风险取舍、登出失效策略、密码重置后旧 token 是否作废。对本地小规模系统可以简化，但 PRD 最好显式写出选择。

**结论**

这份 PRD 的页面层和功能分区已经比较完整，主要问题不在“少画几个页面”，而在“若按文档直接开工，数据层和状态机有几处会卡死实现”。最优先先修 4 件事：

1. 定死艾宾浩斯状态机，只保留一种调度策略。
2. 补完整题库 schema 和导入格式。
3. 统一导入接口与权限边界。
4. 清理所有没有后端闭环的数据按钮，或补数据模型。
